Exam Objectives — Section 6.3
Everything your professor said will be tested
All required knowledge points
  • AIMD: Additive increase (linear growth) + multiplicative decrease (÷2 on loss). Know why the combination matters — and why additive decrease would be insufficient when the network is severely congested.
  • The AIMD sawtooth: CW grows linearly, drops sharply on loss, repeats. This oscillation is normal behavior, not a failure.
  • Slow start — window starts at 1 MSS and doubles each RTT (exponential growth).
  • Slow start ends when CW reaches ssthresh → switch to additive increase.
  • After a loss: new ssthresh = CW/2, restart slow start from 1 MSS.
  • Trace CW through several RTTs of slow start and additive increase — you may need to compute exact values.
  • Why "slow": slow compared to immediately sending at the previous rate (which would burst a full window into the network), NOT slow compared to additive increase.
  • Fast retransmit: 3 duplicate ACKs trigger retransmission without waiting for a timeout. Why 3 and not 1: a single duplicate ACK can be caused by normal packet reordering.
  • Fast recovery: after fast retransmit, halve CW and enter additive increase — do NOT return to slow start. Reasoning: receiving duplicate ACKs means segments are still arriving, so the network is not as severely congested as a timeout suggests.
  • Full scenario trace: slow start → transition to AIMD → loss event → compute new ssthresh and new CW values. This is a likely exam question.
  • CUBIC — why standard AIMD is too slow for high-bandwidth, high-latency links: compute BDP and how many RTTs AIMD needs to recover after a loss.
  • CUBIC's three phases: concave growth (far below W_max), plateau (near W_max where last loss occurred), convex probing (beyond W_max).
  • β = 0.7 in CUBIC vs 0.5 in standard TCP — less capacity surrendered on each loss event.
  • Time-based adjustment (not per-ACK) — why this makes CUBIC fairer between short-RTT and long-RTT competing flows.
The one-page mental model
Effective window MaxWindow = MIN(CongWin, AdvWin)
EffWin = MaxWindow − (BytesSent − BytesAcked)

CongestionWindow is the sender's self-imposed limit based on perceived network capacity. It works alongside the receiver's AdvertisedWindow — the sender respects whichever is smaller.

Two loss signals — different responses
Timeout → Severe congestion

3 Dup ACKs → Milder congestion
ssthresh = CW/2. CW = ssthresh. Fast recovery → AIMD immediately. Pipe still has ACKs — network not fully congested.

AIMD & The Sawtooth
Why additive increase + multiplicative decrease — not any other combination
Additive increase
  • When no loss is detected, CW grows by +1 MSS per RTT. This is linear (additive) growth.
  • Per-ACK increment: Increment = MSS × (MSS / CW). Each ACK adds a fraction; a full window's worth of ACKs sums to +1 MSS.
  • Rationale: cautiously probe for more bandwidth. Overshoot slowly so the network can absorb the ramp-up.
Multiplicative decrease
  • On a timeout (packet loss): CW is halved. Pattern: 16 → 8 → 4 → 2 → 1 MSS. Never below 1 MSS.
  • TCP interprets a timeout as congestion — dropped packets are almost always due to router buffers overflowing, not transmission errors (which are rare on modern links).
  • Halving immediately cuts the sender's contribution to congestion in half.
Why not additive decrease?
The exam will test this reasoning
When the network is severely congested, an oversized window causes dropped packets which are then retransmitted, adding even more traffic — a compounding effect. The sender must exit this state quickly. Additive decrease (−1 MSS per timeout) would take many timeouts to cut a large window; meanwhile congestion compounds. Multiplicative decrease (÷2) cuts the window in half instantly — dramatically reducing load in a single step. This aggressive retreat is what makes the system stable.
The AIMD sawtooth — normal, not a bug
  • Rising slope = additive increase — sender probing for unused capacity, 1 MSS/RTT.
  • Vertical drop = multiplicative decrease — sender reacts to detected congestion.
  • This sawtooth repeats forever. It is the steady-state behavior of a TCP connection in AIMD mode. The oscillation itself is not a problem — it reflects the sender continuously probing and backing off.
Why not AIAD (additive increase / additive decrease)?
AIMD is proven to be a necessary condition for a congestion-control mechanism to be stable. AIAD (−1 on loss, +1 on success) is too symmetric — it takes far too long to reduce load when the network is under severe stress. The asymmetry of AIMD is the key to stability.

Slow Start
Everything your professor listed — in exam-ready detail
The mechanics — step by step
Slow start growth rule Start: CW = 1 MSS
Each ACK: CW += 1 MSS
Net effect: CW doubles per RTT (1 → 2 → 4 → 8 → 16 → …)

Transition rule if CW < ssthresh: exponential growth (slow start)
if CW ≥ ssthresh: linear growth (additive increase, +1/RTT)
  • Window starts at 1 MSS — one packet sent, wait for ACK.
  • Each ACK adds 1 MSS: 1 packet → ACK → 2 packets → 2 ACKs → 4 packets. Doubles each RTT.
  • Ends when CW = ssthresh: switch to additive increase (+1 MSS/RTT) for cautious probing.
  • On timeout during slow start: ssthresh = CW / 2; CW reset to 1; slow start restarts toward the new ssthresh.
Why it's called "slow" — the exact reasoning your professor wants
The comparison that matters
Slow start is exponential and therefore faster than additive increase. But compare it to the original TCP behavior: sending an entire advertised window all at once on connection start. That full-window burst would dump potentially thousands of packets into the network simultaneously, overwhelming router buffers. Slow start is "slow" because it starts with 1 MSS and grows gradually — slow relative to that full-window burst, not relative to additive increase.

Key pattern to memorize
After a loss: ssthresh = CW/2 (round down if needed). CW = 1. Slow start races back to ssthresh exponentially, then additive increase takes over. The new ssthresh is always half the window size at the time of loss.
Two situations where slow start runs
1. Connection start
No information about network capacity. ssthresh is effectively infinite. Slow start runs until the first loss, which sets ssthresh and begins the AIMD cycle.
2. After a coarse-grained timeout
Pipe is empty — no packets in flight, no ACKs to clock. Slow start restarts from CW=1, but now ssthresh gives a target to stop at before switching to AIMD.
When slow start is SKIPPED
After fast retransmit (3 duplicate ACKs), TCP uses fast recovery — it does NOT restart slow start. It goes directly to additive increase at CW = ssthresh. Slow start only runs at connection start and after a coarse-grained timeout.

Fast Retransmit & Fast Recovery
What they are, why 3 ACKs, and the key reasoning difference from a timeout
Fast retransmit — what and why
  • Trigger: 3 duplicate ACKs received at the sender.
  • Action: retransmit the missing packet immediately — without waiting for the retransmission timeout (which could be hundreds of milliseconds).
  • Why 3, not 1: A single duplicate ACK can arise from normal packet reordering — packets arriving out of order. One dup ACK is not reliable evidence of loss. Three is much more likely to indicate a genuinely dropped packet, not just reordering.
How duplicate ACKs arise
When packet N is lost but packets N+1, N+2, N+3 arrive, the receiver sends ACK(N-1) for each one — it can't ACK N because it's missing. The sender sees multiple copies of the same ACK. Three copies = three out-of-order packets arrived = strong evidence that N was dropped.
Fast recovery — the key difference from a timeout
After timeout
No packets in flight. No ACKs arriving. The pipe is empty — the network appears completely congested. Must restart slow start from CW=1.

ssthresh = CW/2, CW = 1.
After 3 dup ACKs (fast recovery)
Duplicate ACKs are still arriving — some segments are getting through. The network is not severely congested. Skip slow start, go directly to AIMD.

ssthresh = CW/2, CW = ssthresh.
The reasoning your professor highlighted
Receiving duplicate ACKs means segments are still getting through to the receiver — duplicate ACKs are only generated when later packets arrive. The network is congested enough to drop one packet but not so congested that nothing is flowing. Therefore, a mild response (halve and continue AIMD) is appropriate rather than the severe response (restart slow start from 1) used after a timeout.

TCP CUBIC — Section 6.3.4
Why standard AIMD fails on fast long-distance links — and how CUBIC fixes it
Why standard AIMD is too slow — the concrete problem

Standard AIMD adds exactly 1 MSS per RTT during additive increase. On a high-bandwidth, high-RTT link this is painfully slow. After a loss event cuts the window in half, it takes an enormous number of RTTs to recover — the sender leaves most of the pipe unused for a long time.

Worked example — be able to compute this
Link: 10 Gbps    RTT: 100 ms
BDP = 10×10⁹ bits/s × 0.1 s = 1×10⁹ bits = 125 MB
Segments to fill pipe ≈ 125 MB / 1500 B ≈ 83,333 MSS

After a loss: window drops to ~41,667 MSS (÷2)
RTTs to recover at +1 MSS/RTT: ~41,667 RTTs
Time to recover: 41,667 × 0.1 s ≈ 4,167 seconds ≈ 70 minutes
The bottleneck
AIMD was designed for the 1980s Internet — small pipes, short RTTs. At 10 Gbps with a 100 ms RTT, a single loss event followed by additive increase means the link sits at half capacity for over an hour. These networks are called long-fat networks (high delay × bandwidth product). CUBIC was designed specifically to solve this.

CUBIC's two key innovations
Innovation 1: Cubic window function
Window grows as a cubic function of time since the last loss event — not as a linear function of ACKs. Three phases: fast growth, plateau near W_max, then aggressive probing beyond W_max.
Innovation 2: Time-based (not ACK-based)
Updates happen on a real-time clock, not per ACK. Standard AIMD grows faster for short-RTT flows (they get ACKs more often). CUBIC's time-based growth equalizes competition between flows with different RTTs.
The three phases of CUBIC — know all three for the exam
Phase 1: Concave
CW is far below W_max. Growth is fast and accelerating — the curve is concave (curving upward). The sender is well below the last congestion point, so it is safe to grow aggressively.
Phase 2: Plateau
CW is near W_max — the window where the last loss occurred. Growth nearly stalls. The sender is cautious: this is where congestion was previously detected, so it probes slowly.
Phase 3: Convex
CW is beyond W_max. Growth accelerates again — the curve is convex (curving downward in the slope). The sender is probing whether new capacity has become available since the last loss.
Why this shape is clever
The concave phase lets CUBIC recover quickly from a loss. The plateau prevents overshooting right where the network last complained. The convex phase aggressively probes for newly available capacity — important on links where available bandwidth changes over time. Standard TCP's additive increase is only convex (always the same slope), so it recovers slowly and never probes aggressively.
β = 0.7 vs β = 0.5 — practical effect
Standard TCP (β = 0.5)
On loss: CW = CW × 0.5 = CW / 2. Surrenders half of the window. Aggressive reaction that may over-reduce capacity on a link that isn't fully congested.
CUBIC (β = 0.7)
On loss: CW = CW × 0.7. Surrenders 30% (keeps 70%). Less capacity is given up per loss event. Combined with faster recovery, CUBIC maintains higher throughput on high-BDP links.
Example
CW = 100,000 MSS (realistic on a 10 Gbps link). On loss:
Standard TCP: CW → 50,000 MSS. Recovers at +1 MSS/RTT — takes 50,000 RTTs (hours).
CUBIC: CW → 70,000 MSS. Recovers via cubic function — reaches W_max in far fewer RTTs, then probes beyond it.
Time-based adjustment and RTT fairness

Standard AIMD adds +1 MSS per RTT. This means a flow with RTT = 10 ms gets ACKs 10× faster than a flow with RTT = 100 ms — and grows its window 10× faster. On a shared bottleneck link, the short-RTT flow dominates.

How CUBIC fixes this
CUBIC updates CongestionWindow based on elapsed wall-clock time since the last congestion event — not on ACK arrivals. A flow with RTT = 10 ms and a flow with RTT = 100 ms will both grow their windows at the same rate in real time (since the cubic function advances the same amount per second). This makes CUBIC significantly fairer when flows with different RTTs compete on the same bottleneck.
