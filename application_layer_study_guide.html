Exam Objectives
All required knowledge points from your professor's guide
HTTP
  • Non-persistent vs persistent: non-persistent opens a new TCP connection per object (costs 2 RTTs per object + slow-start); persistent reuses the connection (saves TCP handshake overhead and slow-start penalty).
  • HTTP is stateless by design — simpler servers, easier scaling. Cookies add state: server sets a cookie in response, browser sends it back on subsequent requests.
  • Third-party cookies enable cross-site tracking — ad network's cookie follows you across every site that loads its ads.
  • Web cache (proxy): reduces response time (closer to client) and bandwidth usage (less traffic on access link). Conditional GET: cache sends If-Modified-Since header; server replies with new object or 304 Not Modified.
  • HOL blocking in HTTP/1.1: responses delivered in order — slow object blocks everything behind it. HTTP/2 solves with multiplexing (interleaved frames). HTTP/3 uses QUIC over UDP to eliminate TCP-level HOL blocking.
Email
  • Full path: sender's mail client → sender's mail server (SMTP push) → recipient's mail server (SMTP push) → recipient's mail client (IMAP/POP pull).
  • SMTP hops are push; IMAP/POP retrieval is pull.
  • Why a recipient mail server is necessary: the recipient's device may not always be online — the server stores messages until the client connects.
DNS
  • Why not centralized: single point of failure, traffic volume, distance, maintenance.
  • Three-level hierarchy: root servers → TLD servers → authoritative servers.
  • Iterative vs recursive: in iterative, the resolver contacts each level itself; in recursive, each server contacts the next on the resolver's behalf. Root/TLD prefer iterative to limit their load.
  • DNS caching + TTL: cached entries expire per TTL — controls freshness vs load tradeoff. DNS changes take time to propagate because all cached copies must expire first.
ICMP + IPv6 / NAT
  • ICMP: diagnostic and error reporting only — not user data. ping uses ICMP Echo Request/Reply to measure round-trip time. traceroute sends packets with incrementing TTL values, triggering ICMP Time Exceeded from each router.
  • IPv6 solves IPv4 address exhaustion: 32-bit (4.3B addresses) → 128-bit (3.4×10³⁸ addresses).
  • NAT as IPv4 workaround: many internal hosts share one public IP; NAT device rewrites port numbers. NAT violates end-to-end principle — breaks applications requiring inbound connections (P2P, servers).
  • Dual-stack transition: devices run both IPv4 and IPv6 simultaneously.
  • Why adoption is slow: IPv4 + NAT works well enough for most consumers — no pressure to switch.

HTTP
HTTP — Persistent vs Non-Persistent
What each saves and the cost of non-persistent connections
Non-persistent vs persistent — side by side
Non-Persistent HTTP/1.0
New TCP connection opened per object. Each object costs:
• 1 RTT for TCP handshake (SYN/SYN-ACK)
• 1 RTT for HTTP request/response
• Transmission time
= 2 RTT + transmission time per object

A page with 10 images = 11 TCP connections = 22+ RTTs.
Persistent HTTP/1.1
Connection stays open after sending response. Multiple objects over one TCP connection. Subsequent requests: ~1 RTT per object (no new handshake).

Same page with 10 images ≈ 1 RTT (initial) + small per-request overhead.
What persistent HTTP saves
TCP handshake overhead: no SYN/SYN-ACK for every object.
TCP slow-start penalty: non-persistent connections restart slow-start (CW=1) on each new connection, wasting time building up throughput. Persistent connections avoid this for subsequent objects on the same connection.
HTTP is stateless by design — and why cookies exist
  • Stateless: the server keeps no information about previous requests. Each HTTP request is independent.
  • Benefits of statelessness: simpler server design — no state to maintain, synchronize, or recover if clients crash. Easier to scale horizontally (any server can handle any request).
  • Problem: many applications need to identify returning users (shopping carts, login sessions, personalization). Cookies solve this.
  • How cookies work: (1) Server sends Set-Cookie: 1678 in HTTP response. (2) Browser stores cookie. (3) Browser sends Cookie: 1678 header on every subsequent request to that server. (4) Server looks up cookie ID in its database to identify the user.

Cookies, Third-Party Tracking & Web Caching
How state gets added to stateless HTTP — and conditional GET
Third-party cookies and cross-site tracking

When you visit nytimes.com, the page may load an ad from AdX.com. Your browser fetches that ad directly from AdX.com, which can set its own cookie. Next time you visit socks.com (which also uses AdX), your browser sends the same AdX cookie — and AdX now knows you visited both sites.

Why third-party cookies enable cross-site tracking
The cookie is set by a domain you never chose to visit (AdX.com). But because your browser automatically sends that cookie on every request to AdX.com — regardless of which site you're on — AdX can build a profile of your browsing across every site that loads its content. You don't have to visit AdX.com directly; it tracks you invisibly through embedded resources on other pages.
First-party vs third-party cookies
First-party: cookie from the site you chose to visit — used for login, shopping carts, preferences. Generally expected and necessary.
Third-party: cookie from a domain you did not choose to visit — the ad or tracking network. Enables cross-site profiling. Disabled by default in Firefox and Safari; being phased out in Chrome.
Web caches (proxy servers) — how they reduce load
  • Reduces response time: cache is physically closer to the client than the origin server. A cache hit means near-instant response instead of crossing the access link.
  • Reduces bandwidth usage: cached objects are served locally; traffic never goes out on the institution's (expensive) access link to the Internet.
  • A cache hit rate of 40% reduces access link utilization from 97% (near-saturated) to ~58% — dramatically cutting queueing delay.
Conditional GET — interactive

The cache uses If-Modified-Since to check if its cached copy is still fresh, avoiding unnecessary data transfer.

What each response means
304 Not Modified: the object on the server is unchanged since the date in the request. The cache's copy is still valid — use it. No object data is transmitted, saving bandwidth and time.

200 OK + data: the object was modified after the cached date. The server sends the new version. The cache updates its copy and the TTL.

HTTP/2, HTTP/3 & Head-of-Line Blocking
HOL blocking, multiplexing, and why TCP motivated QUIC
HTTP version progression — click each to expand
HTTP/1.0
1996
Non-persistent connections
One TCP connection per object. Each object costs 2 RTTs + transmission time. Getting a page with 10 images opens 11 separate TCP connections. Extremely inefficient on modern pages with hundreds of objects.
HTTP/1.1
1997
Persistent connections + pipelining → but HOL blocking
Keeps TCP connection open; multiple objects can be requested. BUT: server must respond in order (FCFS). If object 1 is a large video file, objects 2/3/4 (small icons) must wait behind it — this is head-of-line (HOL) blocking. Browsers opened 6+ parallel TCP connections as a workaround.
HTTP/2
2015
Frame multiplexing — solves application-layer HOL blocking
Objects are divided into frames. Frames from different objects are interleaved — small objects (O2, O3, O4) can be delivered in frames between frames of large O1. No need for in-order delivery at the HTTP layer. BUT: still over one TCP connection — a single dropped TCP segment stalls ALL streams until it's retransmitted. This is TCP-level HOL blocking.
HTTP/3
2022
QUIC over UDP — eliminates TCP-level HOL blocking
HTTP/3 moves from TCP to QUIC (a reliable transport built on UDP). Each stream is independent — a dropped packet for one stream doesn't stall other streams. Also adds TLS 1.3 security by default. Because UDP is the base, QUIC implements its own reliability, congestion control, and multiplexing per-stream.
Head-of-line (HOL) blocking — the key concept
In HTTP/1.1: server sends responses in request order. A slow or large object blocks all smaller objects behind it — even if those smaller objects are already ready. This is analogous to being stuck behind a slow truck in a single lane — you can't pass even though you're ready. HTTP/2 fixes this at the HTTP layer by interleaving frames, but a single dropped TCP packet still stalls all frames until retransmission — TCP doesn't know which frame belongs to which stream.

Push vs Pull — the critical distinction
SMTP (push): the sender's server initiates the connection and pushes the message to the recipient's server. Like physically delivering a letter.

IMAP/POP (pull): the recipient's mail client connects to the recipient's server and pulls messages down. The client controls when it retrieves mail.
Why the recipient's mail server is necessary
Bob's phone or laptop may be offline (asleep, out of coverage, powered off). If Alice sent email directly to Bob's device, the delivery would fail whenever he's offline. The recipient's mail server acts as a persistent intermediary — always on, always reachable — that stores messages until Bob's client connects to retrieve them.

DNS
DNS — Hierarchy, Queries & Caching
Why centralization fails and how iterative vs recursive differ
Why not a single centralized DNS server?
Single point of failure
If the server goes down, DNS fails for the entire Internet. No redundancy.
Traffic volume
Comcast DNS alone handles 600B queries/day. A single server cannot handle all Internet DNS traffic.
Distance
A server in one location is far from most of the world — high latency for billions of users.
Maintenance
Billions of records, updated constantly. No single organization can keep up with the entire Internet.
The three-level hierarchy — click each to expand
R
Root Name Servers
13 logical servers, ~200 physical replicas worldwide
Contact of last resort. Root servers know the addresses of all TLD servers. If your local DNS can't find an answer, it asks the root. Root servers return the address of the appropriate TLD server — they don't have individual hostname records. ICANN manages the root DNS domain.
TLD
Top-Level Domain (TLD) Servers
.com, .org, .edu, .net, country codes (.uk, .cn…)
TLD servers know the addresses of authoritative DNS servers for all domains under their TLD. For example, the .com TLD server knows where to find amazon.com's authoritative DNS server. Network Solutions manages .com and .net; Educause manages .edu.
A
Authoritative DNS Servers
Organization-owned, provides final IP mappings
The authoritative server for a domain contains the actual hostname-to-IP mappings (A records). amazon.com's authoritative DNS server knows the IP address of www.amazon.com. Maintained by the organization itself or a DNS hosting provider.
Iterative vs recursive queries

Iterative (preferred by root/TLD)

Local DNS → Root: "Who is cs.umass.edu?"
Root → Local DNS: "Ask the .edu TLD"
Local DNS → TLD: "Who is cs.umass.edu?"
TLD → Local DNS: "Ask umass.edu auth server"
Local DNS → Auth: "Who is cs.umass.edu?"
Auth → Local DNS: "128.119.240.10" ✓

The local DNS resolver does all the work. Root and TLD just redirect, not do the work themselves.

Recursive (heavier for upper levels)

Local DNS → Root: "Who is cs.umass.edu?"
Root → TLD: "Who is cs.umass.edu?"
TLD → Auth: "Who is cs.umass.edu?"
Auth → TLD: "128.119.240.10"
TLD → Root: "128.119.240.10"
Root → Local DNS: "128.119.240.10" ✓

Each server contacts the next on the requester's behalf. Heavy load at root/TLD — this is why they prefer iterative.

DNS caching and TTL
  • Any name server that learns a mapping caches it. Future queries for the same name are answered from cache — no need to traverse the hierarchy.
  • Cached entries expire after their TTL (Time To Live) — a value set by the authoritative server and included in DNS responses.
  • Why DNS changes take time to propagate: when you update an IP address for a domain, old cached entries across the Internet don't disappear immediately. Every cached copy must wait for its TTL to expire before fetching the updated record. With a TTL of 24 hours, a change can take up to 24 hours to be visible everywhere.
TTL tradeoff
Short TTL → fresher data, faster propagation of changes, but more DNS queries (more load on authoritative servers).
Long TTL → stale data possible, but reduces load on authoritative servers significantly. Most TLD entries are cached in local name servers to reduce load on root and TLD servers.

ICMP
ICMP — Ping & Traceroute
Diagnostic tool — not user data
What ICMP is and is not
ICMP is for diagnostics and error reporting only
ICMP (Internet Control Message Protocol) carries diagnostic messages and error reports between network devices — it does not carry application data. It lives at the network layer (same as IP) but is not a transport protocol. Key ICMP message types: Echo Request/Reply (ping), Time Exceeded (traceroute), Destination Unreachable.
How ping works
  • Sender sends an ICMP Echo Request to the target IP address.
  • Target replies with an ICMP Echo Reply.
  • Sender measures the elapsed time = round-trip time (RTT).
  • Multiple probes are sent; ping reports min/avg/max RTT and packet loss.
How traceroute works — interactive simulation

Traceroute sends packets with increasing TTL values. Each router that decrements TTL to 0 sends back an ICMP Time Exceeded message, revealing its IP and RTT.

Click "Run traceroute" to start...
The TTL trick explained
Traceroute sends probe packets with TTL=1, TTL=2, TTL=3, etc. Each router decrements TTL by 1 before forwarding. When TTL reaches 0, the router discards the packet and sends back an ICMP Time Exceeded message with its own source IP. This reveals each hop along the path. * * * means that hop didn't reply (firewall blocking ICMP or no reply).

IPv6 & NAT
IPv6, NAT & the Transition Problem
Address exhaustion, the end-to-end violation, and why IPv6 adoption is slow
IPv4 address exhaustion — the fundamental problem
2³²
IPv4: ~4.3 billion addresses
IANA exhausted its pool in 2011. Regional registries ran out by 2019. Not enough for all devices on Earth.
2¹²⁸
IPv6: ~3.4 × 10³⁸ addresses
Enough to assign billions of addresses to every atom on Earth's surface. Address exhaustion is solved permanently.
NAT — how it works and why it breaks the end-to-end principle

NAT (Network Address Translation) lets many devices share one public IP address by rewriting port numbers:

Both internal hosts appear as the same public IP. NAT table maps (public IP, port) back to (internal IP, port) for return traffic.
Why NAT violates the end-to-end principle
The end-to-end principle says the network should transport packets without modifying them — intelligence lives at the endpoints. NAT violates this by:

1. Rewriting IP headers and transport-layer port numbers inside the network — not at endpoints.
2. Breaking inbound connection initiation: an external host cannot initiate a connection to an internal host behind NAT, because the internal host has no public IP. This breaks P2P applications, hosting servers, VoIP, games — anything requiring inbound connections.
3. Breaking certain protocols that embed IP addresses in the data payload (FTP, SIP) — NAT can't see or update those embedded addresses.
IPv6 transition — dual-stack and why adoption is slow
  • Dual-stack: devices run both IPv4 and IPv6 simultaneously. They use IPv6 when the destination supports it, fall back to IPv4 otherwise. This avoids a hard cutover.
  • Why adoption has been slow: IPv4 + NAT works well enough for typical consumer use. Consumers can browse, stream, and use apps without IPv6 — there's no immediate pain driving adoption.
  • The chicken-and-egg problem: ISPs won't deploy IPv6 until customers demand it; customers don't demand it because everything works with IPv4/NAT.
