§8.1 Trust and Threats
  • CIA Triad: Confidentiality, Integrity, and Availability — the three goals of network security.
  • Trudy model (network intruder capabilities): eavesdropping, modification, injection, replay, and dropping messages in transit.
§8.2 Cryptographic Building Blocks
  • Symmetric key: same key encrypts and decrypts, fast, but key distribution is the challenge.
  • Public key: key pair (public + private), no shared secret needed, but much slower.
  • Public key — confidentiality: encrypt with recipient's public key. Authentication/integrity: sign with sender's private key.
  • Session key: why real systems combine both — public key crypto to exchange a session key, symmetric crypto for the actual data (performance reason).
  • Message digest (cryptographic hash): must be deterministic, one-way, and collision-resistant.
  • TCP/UDP checksum ≠ cryptographic hash — easy to find collisions, does not resist intentional modification.
  • Digital signatures are applied to the hash of a message, not the full message.
§8.3 Key Predistribution and CAs
  • Man-in-the-middle attack on naive public key exchange: Trudy intercepts Alice's and Bob's public keys and substitutes her own.
  • Certification Authority (CA): binds an identity to a public key by issuing a signed certificate.
  • Verifying a certificate: check the CA's digital signature using the CA's public key.

Section 8.1
Trust, Threats & the CIA Triad
The three security goals and the five things a network intruder can do
The CIA Triad — click each to expand
C
Confidentiality
Only authorized parties can read the message.
Achieved through encryption. An eavesdropper who intercepts the ciphertext cannot recover the plaintext without the key. Example: TLS encrypting your credit card number in transit.
I
Integrity
The message was not modified in transit.
Achieved through message authentication codes (MACs) or digital signatures. Even if an attacker can modify ciphertext, integrity checks detect tampering. Example: HMAC appended to a message.
A
Availability
The service is accessible to legitimate users.
Threatened by denial-of-service (DoS) attacks, which flood a target with traffic so legitimate users cannot get through. Confidentiality and integrity tools do not protect availability — separate mechanisms are needed.
Why all three matter together
Security requires all three. A system that is confidential but not available (DDoSed) is broken. A system that is available and confidential but not integral (message tampered) is dangerous. Encryption alone does not provide integrity — an attacker can randomly modify ciphertext and produce valid-looking plaintext.
The Trudy Model — five capabilities of a network intruder

"Trudy" is the conventional name for an adversary who has access to the network. Unlike a passive observer, Trudy can do all of the following:

Eavesdrop Read packets in transit. Trudy is on the path between Alice and Bob and can copy any packet she sees. Threatens confidentiality.
Modify Change the content of a packet before forwarding it. Bob receives a message that Alice never sent. Threatens integrity.
Inject Create and send entirely new packets claiming to be from Alice. Trudy impersonates a legitimate participant. Threatens authentication.
Replay Record a valid packet from Alice and send it again later. Example: replay a "transfer $100" message. Threatens freshness/integrity.
Drop Discard packets silently. Alice thinks Bob received her message; Bob never did. Threatens availability and reliability.
Why this matters
Security mechanisms must be designed against a Trudy-capable adversary, not just passive eavesdroppers. Encryption alone only stops eavesdropping — it does not prevent modification, injection, replay, or dropping.
Trust and threat — the core tradeoff

Security is a cost/benefit calculation. Every mechanism adds overhead (latency, CPU, complexity). You must identify which threats to mitigate and which to accept based on:

  • The value of the data being protected.
  • The probability of an attack.
  • The cost of the countermeasure (performance, complexity, money).
Historical context
The Internet was designed assuming all actors inside the network were trusted. Security was an afterthought — most tools are retrofitted. A comprehensive redesign would place security at the core from the start.

The direction rule

Section 8.2
Session Keys — Combining Both Approaches
Why real systems use public key to set up, then symmetric key for data
The performance problem with pure public key crypto

Public key algorithms (RSA, etc.) are orders of magnitude slower than symmetric algorithms (AES). On a modern CPU, AES can encrypt gigabits per second; RSA can encrypt only thousands of bits per second. Using RSA to encrypt an entire HTTPS session would be impractically slow.

The hybrid solution
Use public key cryptography only for the key exchange phase — to securely transmit a randomly generated session key. Then use that session key with a fast symmetric algorithm (AES) for all the actual data. This combines the best of both worlds: the security of public key for key distribution, and the speed of symmetric key for bulk data.
Session key protocol — how it works
TLS-style handshake (simplified)
Alice
client
1. Alice generates random session key S
2. Encrypts S with Bob's public key
3. Sends encrypted S
Bob
server
Bob decrypts with his private key → both now share session key S
Alice
All data encrypted with AES using session key S
(fast symmetric encryption)
Bob
  • Why a session key? Limits the amount of ciphertext encrypted under any one key. If a key is compromised, only one session is exposed. Fresh key for each session = forward secrecy.
  • Why symmetric for data? AES is thousands of times faster than RSA. Encrypting gigabytes of video with RSA is not feasible.
  • Why public key for the handshake? Alice and Bob can establish the session key without ever having met — no pre-shared secret needed.

Section 8.2
Hashes, MACs & Digital Signatures
Message digests, why checksums aren't enough, and why we sign the hash
Cryptographic hash — three required properties
Deterministic
The same input always produces the same output. hash("hello") is always the same value.
One-way
Computationally infeasible to find any input that produces a given output. Cannot reverse the hash to find the plaintext.
Collision-resistant
Computationally infeasible to find two different inputs that produce the same output. Finding any collision should require ~2⁶⁴ attempts.
What a hash does
A cryptographic hash function takes an arbitrary-length message and produces a fixed-length digest (e.g., SHA-256 always produces 256 bits). The digest is like a fingerprint of the message — any change to the message produces a completely different digest.
Why TCP/UDP checksum ≠ cryptographic hash
TCP/UDP checksum
Simple sum of 16-bit words. Designed to catch accidental bit errors. An adversary can easily modify a message and adjust the checksum to match — trivial to find collisions. Not a cryptographic tool.
Cryptographic hash (SHA-256)
Designed to resist deliberate modification. Finding any other message with the same digest requires ~2⁶⁴ attempts even knowing the hash function. An adversary cannot craft a fake message with a matching digest without the key.
The key difference
TCP checksum detects random errors (bit flips from noisy links). A cryptographic hash prevents intentional, adversarial modification. A determined attacker can trivially forge a TCP checksum; forging a SHA-256 HMAC is computationally infeasible.
Why digital signatures are applied to the hash, not the full message
  • Public key operations are slow. RSA encrypting a 10 MB file directly is computationally expensive — could take seconds.
  • The hash is always a fixed small size. SHA-256 always produces 32 bytes regardless of input. Signing 32 bytes with RSA is fast.
  • The hash binds the signature to the content. Signing the hash of the message is equivalent to signing the message itself — any change to the message changes the hash, invalidating the signature.
Digital signature process
Sender:  hash(message) → digest → sign(digest, private_key) = signature
Transmit: message + signature
Receiver: hash(received_message) → digest′  |  verify(signature, sender_public_key) → digest
Check: digest′ == digest → authentic & unmodified
HMAC — keyed hash for message authentication

An HMAC (Hash-based Message Authentication Code) produces an authentication tag that is a function of both the message and a secret key. Only someone with the key can produce or verify the HMAC.

HMAC formula HMAC = H((K ⊕ opad) || H((K ⊕ ipad) || message))
HMAC vs digital signature
Both authenticate messages, but HMACs use symmetric keys (fast, no nonrepudiation — both parties know the key). Digital signatures use asymmetric keys (slower, but provide nonrepudiation — only the private key holder could have signed). Use HMAC for performance; use digital signatures when you need a third party to be able to verify.

Section 8.3
CAs, Certificates & Man-in-the-Middle
Why you can't just publish your public key — and how CAs solve it
The man-in-the-middle attack on naive public key exchange

Click each step to understand the attack:

1
Alice publishes her public key
Alice generates a key pair and wants to share her public key with Bob so he can send her encrypted messages or verify her signatures. She posts it on a server or sends it directly. But the channel is not authenticated.
2
Trudy intercepts and substitutes her own public key
Trudy sits between Alice and Bob. When Alice's public key is transmitted to Bob, Trudy intercepts it and sends her own public key to Bob instead. Bob now has Trudy's key, believing it is Alice's. Similarly, when Bob sends his public key to Alice, Trudy substitutes her own.
3
Bob encrypts to "Alice" — actually to Trudy
Bob encrypts a message using what he thinks is Alice's public key. Since it's actually Trudy's key, Trudy can decrypt it. Trudy reads the message, re-encrypts it with Alice's real public key, and forwards it. Neither Alice nor Bob know Trudy is in the middle.
4
Result: Trudy reads and can modify all traffic
Trudy has established a session with both Alice and Bob, each of whom believes they are talking directly to the other. Trudy can read, modify, and inject messages into both directions — a complete man-in-the-middle compromise. This is why naive public key exchange without authentication is useless.
The root problem
Public key exchange is only secure if Bob can verify that the public key he received actually belongs to Alice — and not to Trudy impersonating Alice. Without a trusted third party or some authenticated channel, Bob has no way to make this verification.
What a Certification Authority (CA) does
  • A CA is a trusted third party that binds an identity to a public key by issuing a digitally signed certificate.
  • The CA verifies that the entity claiming to own a public key actually controls the corresponding identity (domain name, email, organization).
  • The CA then signs the certificate with its own private key. Anyone who trusts the CA and knows the CA's public key can verify the certificate.
  • Your browser comes pre-loaded with the public keys of dozens of trusted root CAs — this is the basis of HTTPS.
What a certificate contains
X.509 Certificate fields Identity:  alice@example.com (or www.example.com)
Public key:  Alice's actual public key value
CA identity:  VeriSign (the signer)
Signature:  CA's digital signature over the above fields
Sig. algorithm:  RSA with SHA-256
Expiration:  2026-01-01 (optional but common)
How a recipient verifies a certificate — the exact process
1 Bob receives Alice's certificate. It contains: Alice's identity, Alice's public key, and the CA's digital signature over those fields.
2 Bob looks up the CA's public key (which he obtained out-of-band — it may be pre-installed in his browser or OS).
3 Bob verifies the CA's digital signature on the certificate using the CA's public key. If the signature is valid, the certificate was genuinely signed by the CA.
4 Since the CA verified Alice's identity before signing, and the signature is valid, Bob now trusts that the public key in the certificate really belongs to Alice — not Trudy.
Why this stops the MITM attack
Trudy can intercept Alice's certificate, but she cannot forge the CA's signature (she doesn't have the CA's private key). If Trudy sends her own certificate claiming to be Alice, the CA's signature on it will say "Trudy.com" not "Alice.com" — Bob's verification will fail or show the wrong identity.
