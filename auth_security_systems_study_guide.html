§8.4 Authentication Protocols
  • ap1.0 — Identity claim fails: stating your identity is not enough — anyone can claim to be Alice.
  • ap2.0/3.0 — Passwords fail: sending a password over the network is vulnerable to eavesdropping (ap2.0) and replay attacks (ap3.0 — capture the encrypted password and resend it).
  • ap5.0 — Nonce-based challenge-response works: a nonce (one-time random number) defeats replay attacks because each response is computed fresh over a value the attacker has never seen before.
  • Be able to walk through the full nonce-based authentication exchange: Bob sends nonce R → Alice signs or HMACs R → Bob verifies.
§8.5 Example Security Systems
  • TLS — three services: confidentiality (symmetric encryption), integrity (MAC per record), authentication (public key certificates). Know which cryptographic technique provides each.
  • TLS handshake derives multiple keys from a master secret — separate keys for each direction and purpose.
  • Why MAC per record (not per stream): so the receiver can detect tampering immediately on each record without waiting for the entire stream.
  • Stateless packet filtering — examines individual packet headers (IP addresses, ports, TCP flags) with no memory of prior packets.
  • Stateful filtering — tracks connection state, can reject packets that don't belong to an established connection. Stronger than stateless.
  • Reasoning about rules: blocking inbound SYN packets prevents external hosts from initiating connections to internal servers.
  • 802.11 security — four steps: discovery → mutual authentication with Authentication Server (AS) → key distribution → encrypted communication.
  • Why authenticate to the AS, not the AP: the AP is untrusted infrastructure — authenticating to it would let a rogue AP impersonate the network. The AS is a trusted server inside the network.

Section 8.4
Authentication Protocol Progression
From naive identity claim to secure nonce challenge-response
The progression — ap1.0 through ap5.0

Each version fixes the flaw of the previous. Know why each of ap1.0–ap4.0 fails and exactly why ap5.0 works.

ap1.0
✗ Broken
Just state your identity
Alice → Bob: "I am Alice"
Attack: Anyone can send this message. Trudy types "I am Alice" and Bob cannot tell the difference. No proof of identity whatsoever.
ap2.0
✗ Broken
Send a shared secret (password) in plaintext
Alice → Bob: "I am Alice, password = hunter2"
Attack: Trudy eavesdrops and learns Alice's password. She then uses it to authenticate as Alice. Password exposed on every login.
ap3.0
✗ Broken
Encrypt the password
Alice → Bob: "I am Alice" + Encrypt(K_AB, password)
Replay attack: Trudy records the entire encrypted message. Later she replays it verbatim to Bob. Bob cannot distinguish a replay from a fresh login — the ciphertext is identical.
ap4.0
✗ Broken
Use a nonce — but only with symmetric key
Bob → Alice: nonce R
Alice → Bob: Encrypt(K_AB, R)
Works against replay but requires a pre-shared secret key K_AB. Does not scale well — Bob needs a unique shared key with every entity he authenticates. Man-in-the-middle is still possible without authentication of the nonce source.
ap5.0
✓ Secure
Nonce + public key digital signature
Bob → Alice: nonce R (random, never used before)
Alice → Bob: (Alice's certificate) + Sign(Alice's private key, R)
Bob verifies: check cert with CA, then verify signature on R
Why this works: Trudy cannot replay — each R is unique. Trudy cannot forge the signature — she lacks Alice's private key. The certificate proves the signing key belongs to Alice. Fresh, unforgeable, and authenticated.
What makes a nonce powerful
A nonce (Number used ONCE) is a random value generated fresh for each authentication attempt. Because Alice must compute her response over this specific nonce, a captured response from a previous session is useless — it was computed over a different nonce that Bob will never send again. The nonce is the "freshness proof."
Why replay attacks break password-based schemes
The replay attack — step by step
1. Alice authenticates to Bob normally, sending Encrypt(K, password).
2. Trudy records this encrypted message — she doesn't need to decrypt it.
3. Alice logs out. Later, Trudy sends the recorded message to Bob verbatim.
4. Bob decrypts it, gets Alice's password, and lets Trudy in as Alice.

The problem: the encrypted password is the same every time. Bob has no way to know if the message is fresh or a recording from the past. A nonce forces the response to be different every single time.

Section 8.5
TLS — Transport Layer Security
Three services, the handshake, and why MAC per record
The three security services TLS provides
Confidentiality
Mechanism: Symmetric encryption (AES) using session keys derived from the master secret.

After the handshake, all data is encrypted so eavesdroppers see only ciphertext.
Integrity
Mechanism: MAC (HMAC) computed over each individual record.

The receiver verifies the MAC before accepting any record. Tampering is detected immediately without waiting for the stream to end.
Authentication
Mechanism: Public key certificates (verified via CA chain).

During the handshake, the server (and optionally the client) presents a certificate proving their identity.
The TLS handshake — how multiple keys are derived
  • Client and server negotiate cipher suites and exchange random values.
  • A pre-master secret is established (via RSA or Diffie-Hellman key exchange, protected by the server's public key).
  • Both sides independently derive a master secret from the pre-master secret and the random values exchanged earlier.
  • From the master secret, multiple session keys are derived using a pseudorandom function (PRF):
    • Client → Server encryption key
    • Server → Client encryption key
    • Client → Server MAC key
    • Server → Client MAC key
Why separate keys for each direction?
Using the same key in both directions could create vulnerabilities where a message from one direction could be mistaken for a message from the other. Separate keys ensure that client-to-server traffic and server-to-client traffic are cryptographically independent.
Why TLS computes a MAC per record — not over the entire stream
MAC over entire stream (hypothetical)
The receiver must wait until all data arrives before verifying. An attacker could inject or drop records in the middle of a long stream, and the tampering would only be detected at the very end — after the data was already processed.
MAC per record (actual TLS)
The receiver verifies each record's MAC immediately upon receipt. Tampering is detected the moment the corrupted or injected record arrives — before it is processed or passed to the application. Provides immediate, fine-grained integrity checking.
Analogy
Think of per-record MAC like checking the seal on each page of a document before reading it, versus checking only the final page. If someone swaps page 5, per-record checking catches it at page 5. Stream-only checking catches it only after you've read the whole document.

Section 8.5
Firewalls
Stateless vs stateful filtering — and reasoning about rules
Stateless vs stateful packet filtering

Stateless Packet Filtering

  • Examines each packet independently — no memory of previous packets.
  • Makes decisions based on: source IP, destination IP, source port, destination port, TCP flags (SYN, ACK, FIN), protocol (TCP/UDP).
  • Simple and fast. Cannot detect attacks that span multiple packets.
  • Weakness: cannot distinguish an ACK that belongs to an established connection from a spoofed ACK injected by an attacker.

Stateful Packet Filtering

  • Tracks connection state — remembers which connections have been established.
  • Maintains a table of active connections (src IP, src port, dst IP, dst port, TCP state).
  • Can reject packets that don't belong to any established connection — even if headers look valid.
  • Stronger: detects spoofed ACKs, half-open SYN attacks, and out-of-state packets.

Why blocking inbound SYN stops external initiation
A TCP connection always begins with a SYN packet from the initiating host. By blocking inbound SYNs (from external → internal), the firewall prevents any external host from starting a connection to an internal server. Internal hosts can still initiate outbound connections (their SYNs go outbound), and the reply SYN-ACK and ACK packets (which have the ACK flag set and match an existing state entry) are allowed back in.

Section 8.5
802.11 Wireless Security
Four-step secure connection process and why the AS matters
The four-step secure wireless connection process — click each to expand
Step 1
Discovery
The mobile device scans for available networks. The Access Point (AP) broadcasts its SSID (network name) and lists the security capabilities it supports (e.g., WPA2-Enterprise, cipher suites). The mobile selects a network and initiates the association process. No keys have been exchanged yet — this is purely capability discovery.
Step 2
Mutual Authentication with the Authentication Server (AS)
The mobile and the Authentication Server (AS) — a trusted server inside the network, separate from the AP — authenticate each other. This typically uses EAP (Extensible Authentication Protocol) over TLS or similar. The mobile proves its identity to the AS (e.g., via certificate or password). The AS proves its identity to the mobile (via its certificate). The AP forwards EAP messages but cannot read or modify them. Neither side trusts the AP for authentication.
Step 3
Key Distribution
After mutual authentication succeeds, the AS and mobile derive a shared Master Session Key (MSK). The AS securely sends a derived key to the AP. Both the mobile and AP now have a shared key — but the AP never knew the mobile's credentials and the mobile never trusted the AP during authentication. The key is distributed by the trusted AS, not negotiated with the AP.
Step 4
Encrypted Communication
The mobile and AP use the distributed key (via a 4-way handshake defined in 802.11i/WPA2) to derive pairwise temporal keys (PTK) for encrypting all wireless frames. Data is now encrypted using AES-CCMP. Each frame is confidential and integrity-protected. The 4-way handshake also ensures both sides possess the same key before data flows.
Why authenticate to the AS — not directly to the AP
The rogue AP attack — why the AP can't be trusted
An attacker can set up a rogue AP with the same SSID as a legitimate network. If authentication happened directly between the mobile and AP, the mobile would authenticate to the rogue AP — handing its credentials directly to the attacker. The attacker now impersonates the mobile to the real network.
If mobile authenticated to the AP
Rogue AP intercepts credentials. Mobile thinks it is connected to the real network. Attacker has full MITM position. No defense against rogue APs.
Mobile authenticates to the AS
The AS is a trusted server with a certificate. The mobile verifies the AS's certificate before proceeding. A rogue AP cannot impersonate the AS — it doesn't have the AS's private key. Authentication is secure even over untrusted AP infrastructure.
Key insight for the exam
The AP is just a radio relay — it is untrusted infrastructure, just like an Ethernet switch. Trust is placed in the Authentication Server, which is inside the secure network perimeter and holds a verifiable certificate. The AP forwards authentication messages without being able to read or forge them.
