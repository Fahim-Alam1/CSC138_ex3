What Section 6.1 Is About
The framework for understanding all congestion control mechanisms
What you need to know
  • The difference between resource allocation, congestion control, and flow control — these are three distinct concepts often confused.
  • The flow abstraction: what it is, how it differs from a channel, and the idea of soft state.
  • The three taxonomy dimensions: router-centric vs host-centric, reservation-based vs feedback-based, window-based vs rate-based.
  • The power metric: Power = Throughput / Delay, how to read the power curve, what congestion collapse is.
  • Jain's fairness index: the formula, what it outputs, and how to interpret it.
The three terms — not the same thing
Resource Allocation
The process by which network elements meet competing demands for link bandwidth and buffer space. Decides how much each flow gets.
Congestion Control
Preventing or responding to overload in the network. Keeps senders from injecting too much data into a bottleneck link.
Flow Control
Preventing a fast sender from overrunning a slow receiver. End-to-end issue. Not the same as congestion control, though they share mechanisms.
Why this matters

Congestion is bad for everyone — it causes delay, packet loss, and ultimately network collapse. Unlike routing (which finds paths), congestion control manages the rate at which traffic enters the network.

Key insight: routing ≠ congestion control
Even if routing assigns large weights to congested links, some bottleneck routers cannot be routed around. Routing adjusts paths; congestion control adjusts load. They solve different problems.
The dual challenge
Effective allocation requires high throughput AND low delay — but these goals are in tension. Packing the network to maximize throughput causes queueing delay to explode.

Key Definitions
Flows, soft state, and the network model
The flow abstraction

A flow is a sequence of packets sent between a source/destination pair that follows the same route through the network. Unlike a channel (an end-to-end abstraction), a flow is visible to routers inside the network.

Host-to-host flow
Same source and destination IP addresses. Coarser granularity — all traffic between two hosts, regardless of port.
Process-to-process flow
Same (srcIP, srcPort, dstIP, dstPort). Finer granularity — essentially the same as a TCP connection.
Implicit vs Explicit flows
An implicit flow is identified by routers watching packet headers — no setup required. An explicit flow is established by a flow setup message from the source. Explicit flows are not the same as virtual circuits — they don't imply reliable ordered delivery; they exist only for resource allocation.
Soft state vs hard state

Hard state (connection-oriented)

Created and removed by explicit signalling. Resources are definitively reserved. Example: VC table entries in ATM. Advantage: guaranteed resources. Disadvantage: underutilization when the connection is idle.

Soft state (middle ground)

Created implicitly as packets arrive, expires if no traffic. Correct network operation does not depend on it — but when it's present, the router can make better decisions. Combines benefits of both approaches.

Why soft state matters
Pure connectionless (no router state) is maximally flexible but limits resource allocation ability. Pure connection-oriented (hard state) wastes resources when flows are idle. Soft state is the pragmatic middle ground used in many real congestion mechanisms.

Taxonomy of Resource Allocation
Three dimensions × two choices — know all six and how they combine
Dimension 1: Where does control live?
Router-centric
Network does the work
Each router decides when to forward packets, which to drop, and sends advisory messages to hosts. Routers police host behavior. Requires per-flow state at routers.
Host-centric
Endpoints adapt themselves
End hosts observe network conditions (packet loss, delay) and adjust their sending rate. Routers have minimal involvement — they just drop packets when overloaded.
Not mutually exclusive
In practice, both participate. Host-centric systems still rely on routers to drop packets when congested. Router-centric systems still expect hosts to obey advisory messages. The question is where the majority of the burden falls.
Dimension 2: How is capacity communicated?
Reservation-based
Ask before sending
A host requests capacity before sending. Each router checks if it can accommodate the request without overcommitting. Like getting a reservation at a restaurant — or a busy signal when the resource is full.
Feedback-based
Send, then adapt
Hosts start sending without reserving capacity. They receive feedback — either explicit (router sends "slow down") or implicit (packet loss is a signal) — and adjust accordingly.
Important implication
Reservation-based always implies router-centric (routers must track reserved resources). Feedback-based can be either: explicit feedback involves the router; implicit feedback (loss-based) places almost all burden on end hosts.
Dimension 3: How is the allowed sending rate expressed?
Window-based
Bytes outstanding
A window size limits how many bytes can be unacknowledged at a time. Familiar from TCP. Indirectly related to bandwidth — window / RTT ≈ throughput. Used by TCP congestion control.
Rate-based
Bits per second
Directly expresses throughput as bits/sec. Natural for multimedia apps with minimum bandwidth requirements. Common in reservation systems (e.g., "I need 1 Mbps for this video stream").

The Power Metric
Throughput / Delay — reading the power curve
The formula
Power of a network connection Power = Throughput / Delay
  • Goal: maximize power — get high throughput with low delay simultaneously.
  • Increasing load initially increases throughput (links are busier) but eventually causes queuing, which increases delay.
  • The power curve has a peak. To the left: too conservative (underutilized links). To the right: too many packets causing queueing delay to dominate.
  • The ideal operating point is at the peak. Resource allocation mechanisms try to keep load there.

Congestion collapse
If load climbs far to the right of the peak, throughput can fall toward zero while the network remains full of packets — all retransmissions, no useful work. This is called congestion collapse. The goal of any stable congestion-control mechanism is to prevent this.
Limitations of the power metric
  • Based on M/M/1 queuing theory which assumes infinite buffers — real networks have finite buffers and must drop packets.
  • Defined relative to a single flow — unclear how it extends to multiple competing connections.
  • Despite limitations, no widely accepted alternative exists, so power continues to be used as a benchmark.

Fair Resource Allocation
Jain's fairness index — formula, intuition, and calculator
What is fair?

In the absence of explicit reservations, "fair" typically means equal shares — each flow sharing a bottleneck link gets the same bandwidth. But equal shares may not always be truly fair (consider a 4-hop flow competing with a 1-hop flow).

Jain's Fairness Index
Raj Jain proposed a mathematical metric that quantifies fairness regardless of the number of flows. It always produces a value between 0 and 1, where 1 = perfectly fair and values approaching 0 indicate severe unfairness.
The formula
Jain's Fairness Index — given n flows with throughputs x₁, x₂, … xₙ f(x₁, …, xₙ) = (Σxᵢ)² / (n · Σxᵢ²)
Perfect fairness → f = 1
All n flows get the same throughput x. Then: (nx)² / (n · nx²) = n²x² / n²x² = 1.
Only k of n flows active → f = k/n
If k flows get equal throughput and n-k get zero, the fairness index drops to k/n — intuitively capturing that only k/n of the flows are being served.
Key property
Whether a single outlier flow gets more or less than the others (positive or negative Δ), the fairness index drops below 1. Any deviation from equality reduces fairness — the metric is symmetric around equal allocation.
